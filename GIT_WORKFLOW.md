# Работа с Git — инструкция для тестировщика

Эта инструкция описывает полный цикл работы с Git — от начала задачи до её завершения.

---

## Основные понятия (если ты новичок)

- **Репозиторий** — папка проекта которая отслеживается Git
- **Ветка (branch)** — отдельная копия проекта для работы над задачей
- **Коммит (commit)** — сохранённое состояние файлов с описанием что изменилось
- **Push** — отправка изменений на GitHub
- **Pull Request (PR)** — запрос на слияние твоей ветки с основной
- **Merge** — слияние двух веток в одну
- **main** — основная ветка, всегда должна быть рабочей

---

## Полный цикл работы над задачей

### Шаг 1 — Перед началом работы обнови main

Всегда начинай с актуальной версии основной ветки:

```bash
git checkout main
git pull origin main
```

### Шаг 2 — Создай новую ветку для задачи

```bash
git checkout -b feature/название-задачи
```

Примеры названий веток:
```bash
git checkout -b feature/add-link-checker
git checkout -b fix/responsive-report-bug
git checkout -b feature/console-checker
```

Правила именования:
- `feature/` — новая функциональность
- `fix/` — исправление бага
- `hotfix/` — срочное исправление в продакшене
- Используй дефисы, не пробелы
- Пиши по-английски

### Шаг 3 — Работай над задачей

Вноси изменения в файлы как обычно.

### Шаг 4 — Проверь что изменилось

```bash
git status
```

Покажет какие файлы изменены, добавлены или удалены.

```bash
git diff
```

Покажет конкретные изменения в строках кода.

### Шаг 5 — Добавь файлы в коммит

Добавить все изменённые файлы:
```bash
git add .
```

Добавить конкретный файл:
```bash
git add название_файла.py
```

### Шаг 6 — Сделай коммит

```bash
git commit -m "описание что сделано"
```

Примеры хороших описаний:
```bash
git commit -m "add link checker with broken links report"
git commit -m "fix responsive checker false positives on tap zones"
git commit -m "add README with installation instructions"
```

Правила описания коммита:
- Пиши по-английски (общепринятый стандарт)
- Начинай с глагола: add, fix, update, remove, refactor
- Коротко и по делу — что именно сделано
- Не пиши "правки", "изменения", "обновление" — это ни о чём не говорит

### Шаг 7 — Отправь ветку на GitHub

```bash
git push origin feature/название-задачи
```

### Шаг 8 — Открой Pull Request на GitHub

1. Зайди на github.com в свой репозиторий
2. Появится жёлтая плашка — нажми **Compare & pull request**
3. Заполни название PR (обычно то же что и коммит)
4. В описании напиши что сделано и зачем
5. Нажми **Create pull request**

### Шаг 9 — Слей PR с main

На странице PR нажми **Merge pull request** → **Confirm merge**.

### Шаг 10 — Обнови локальный main и удали ветку

```bash
git checkout main
git pull origin main
git branch -d feature/название-задачи
git push origin --delete feature/название-задачи
```

## Частые команды

| Команда | Что делает |
|---|---|
| `git status` | Показывает что изменилось |
| `git branch` | Показывает все локальные ветки |
| `git branch -a` | Показывает все ветки включая удалённые |
| `git checkout main` | Переключиться на ветку main |
| `git checkout -b название` | Создать новую ветку и переключиться на неё |
| `git add .` | Добавить все изменения в коммит |
| `git commit -m "текст"` | Сохранить изменения с описанием |
| `git push origin название` | Отправить ветку на GitHub |
| `git pull origin main` | Подтянуть последние изменения с GitHub |
| `git log --oneline` | Показать историю коммитов кратко |
| `git branch -d название` | Удалить ветку локально |
| `git push origin --delete название` | Удалить ветку на GitHub |

## Правила которые стоит соблюдать

**Никогда не коммить напрямую в main**
Всегда создавай отдельную ветку для каждой задачи. Main должен быть всегда рабочим.

**Один коммит — одно изменение**
Не смешивай несколько несвязанных изменений в один коммит. Лучше сделать три маленьких коммита чем один большой со всем подряд.

**Удаляй ветки после мержа**
Feature-ветки живут только пока идёт работа над задачей. После мержа — удаляй.

**Всегда делай pull перед началом работы**
Чтобы не работать на устаревшем коде и не получить конфликты при мерже.

**Не пушь файлы с ключами**
Файл `.env` с токенами никогда не должен попасть на GitHub. Убедись что в `.gitignore` есть строка `.env`.

## Если что-то пошло не так

**Хочу отменить последний коммит (но оставить изменения в файлах):**
```bash
git reset --soft HEAD~1
```

**Хочу отменить все несохранённые изменения в файле:**
```bash
git checkout -- название_файла.py
```

**Случайно закоммитил не в ту ветку:**
```bash
git checkout нужная-ветка
git cherry-pick хэш-коммита
```

**Посмотреть хэши коммитов:**
```bash
git log --oneline
```

**Конфликт при мерже — страшно но не смертельно:**
Git покажет в каком файле конфликт. Открой файл, найди строки с `<<<<<<`, `=======`, `>>>>>>>`  — они показывают два варианта изменений. Оставь нужный вариант, удали маркеры, сохрани файл и сделай коммит.

## Типичный рабочий день

```bash
# Утро — начинаю новую задачу
git checkout main
git pull origin main
git checkout -b feature/my-task

# Работаю...

# Сохраняю прогресс
git add .
git commit -m "add initial implementation"

# Продолжаю работать...

# Финальный коммит и отправка
git add .
git commit -m "complete feature with tests"
git push origin feature/my-task

# Открываю PR на GitHub, жду ревью, мержу

# Убираю за собой
git checkout main
git pull origin main
git branch -d feature/my-task
git push origin --delete feature/my-task
```
